아키텍처는 한 번 잘 설계했다고 끝나는 것이 아닙니다. 트래픽 패턴이 바뀌고, 새로운 AWS 서비스가 출시되고, 팀 규모가 변하면서 어제의 좋은 설계가 오늘의 비효율이 될 수 있습니다. SAP-C02의 마지막 주제는 "이미 운영 중인 워크로드를 어떻게 정기적으로 점검하고 개선하는가"입니다.

## AWS Well-Architected Tool

**Well-Architected Tool**은 AWS Management Console에서 무료로 사용할 수 있는 자가 진단 도구입니다. 실제 운영 중인 워크로드를 등록하고, 6 Pillars(운영 우수성, 보안, 신뢰성, 성능 효율성, 비용 최적화, 지속 가능성)에 걸친 질문에 답하면, 현재 설계의 위험 요소(High Risk Issue, Medium Risk Issue)를 자동으로 식별해줍니다.

사용 흐름은 다음과 같습니다.

1. **워크로드 정의**: 검토 대상 워크로드의 환경(Production/Pre-production), 관련 계정, 적용할 렌즈(Lens)를 지정
2. **렌즈별 질문 응답**: 각 Pillar의 질문(예: "장애 발생 시 자동으로 복구되는가?")에 대해 현재 상태를 체크
3. **위험 식별**: 답변을 기반으로 위험 항목을 자동 분류하고, 관련 AWS 문서와 권장 사항을 연결
4. **개선 계획(Improvement Plan) 수립**: 우선순위가 높은 위험부터 개선 작업을 백로그에 등록
5. **마일스톤 저장**: 개선 후 재검토하여 위험이 줄었는지 기록하고 시점별로 비교

**Lens**(렌즈)는 표준 Well-Architected Framework 질문 외에 특정 도메인(서버리스, SaaS, 머신러닝, 금융 서비스 등)에 특화된 추가 질문 세트입니다. 워크로드 특성에 맞는 렌즈를 추가로 적용하면 더 구체적인 진단을 받을 수 있습니다.

{{< callout type="info" >}}
**실무 팁**: Well-Architected Tool 리뷰는 한 번으로 끝내지 말고 분기(쿼터) 단위로 정례화하는 것이 효과적입니다. 마일스톤 기능으로 "지난 리뷰 대비 위험이 몇 개 줄었는가"를 추적하면, 팀의 개선 활동을 정량적으로 보여줄 수 있습니다.
{{< /callout >}}

## 정기 리뷰를 운영 프로세스로 만들기

도구를 쓰는 것보다 중요한 것은 이를 조직의 정기 운영 프로세스로 정착시키는 것입니다. 효과적인 패턴은 다음과 같습니다.

- **워크로드 단위 책임자 지정**: 각 워크로드마다 리뷰를 주도할 담당자(보통 해당 서비스의 SA나 리드 엔지니어)를 지정
- **High Risk Issue에 대한 SLA**: 리뷰에서 발견된 High Risk Issue는 다음 스프린트 내 해결을 원칙으로 백로그에 우선 반영
- **신규 워크로드 출시 전 리뷰 의무화**: 프로덕션 배포 전 체크리스트에 Well-Architected 리뷰 통과를 포함

## 운영 우수성 피드백 루프

이 정기 리뷰는 결국 **운영 우수성(Operational Excellence)** Pillar가 말하는 피드백 루프의 실천입니다. 변경을 만들고, 그 결과를 관찰하고(CloudWatch, X-Ray), 무엇이 잘못되었는지 학습하고(사후 분석, Post-incident Review), 다시 설계에 반영하는 순환 구조를 Well-Architected Tool 리뷰가 정기적인 체크포인트로 만들어줍니다.

{{< callout type="warning" >}}
Well-Architected Tool은 진단 도구일 뿐, 발견된 문제를 자동으로 고쳐주지 않습니다. 리뷰 결과를 실제 아키텍처 변경으로 이어지게 하려면 조직의 우선순위 결정 프로세스와 반드시 연결되어야 합니다. 그렇지 않으면 "리뷰는 했지만 아무것도 바뀌지 않는" 형식적 활동이 됩니다.
{{< /callout >}}
