온프레미스 데이터센터를 AWS로 옮기는 작업은 "그대로 복사"부터 "완전히 재설계"까지 스펙트럼이 넓습니다. SAP-C02는 이 스펙트럼 안에서 비즈니스 제약(예산, 기한, 팀 역량)에 맞는 전략을 고르는 판단력을 평가합니다.

## 마이그레이션 6R 전략

AWS가 정의하는 6가지 마이그레이션 전략은 "얼마나 많이 바꿀 것인가"를 기준으로 나뉩니다.

1. **Rehost (Lift-and-Shift)**: 애플리케이션 코드를 거의 바꾸지 않고 EC2로 그대로 이전. 가장 빠르지만 클라우드 네이티브 이점을 충분히 활용하지 못함
2. **Replatform (Lift-Tinker-and-Shift)**: 코드는 유지하되 일부 컴포넌트를 관리형 서비스로 교체. 예: 자체 운영 MySQL을 RDS로 교체
3. **Refactor/Re-architect**: 애플리케이션을 클라우드 네이티브 아키텍처로 재설계. 가장 많은 시간이 들지만 장기적으로 가장 큰 효율을 얻음
4. **Repurchase**: 기존 솔루션을 SaaS로 교체. 예: 자체 CRM을 Salesforce로 교체
5. **Retain**: 당장 이전하지 않고 그대로 유지. 라이선스 제약이나 레거시 의존성이 있는 경우
6. **Retire**: 더 이상 필요 없는 애플리케이션을 폐기

실무에서는 수백 개의 애플리케이션을 한꺼번에 분류해야 하므로, 보통 7R(Relocate 추가)까지 확장해서 사용하기도 하지만 시험에서는 6R 분류를 기준으로 출제됩니다.

{{< callout type="info" >}}
**판단 기준**: "빨리 옮겨야 하는가, 잘 옮겨야 하는가"가 핵심 trade-off입니다. 데이터센터 계약 종료가 임박했다면 Rehost, 장기적으로 운영 비용을 줄이고 싶다면 Replatform이나 Refactor를 우선 검토하는 식으로 시나리오의 제약 조건을 먼저 읽어야 합니다.
{{< /callout >}}

## AWS Migration Hub로 전체 진행 상황 추적

**Migration Hub**는 여러 마이그레이션 도구(Application Migration Service, DMS, Migration Evaluator 등)에서 발생하는 진행 상황을 한 곳에서 모아 보여주는 대시보드입니다. 수백 대의 서버를 이전하는 대규모 프로젝트에서 "지금 몇 대가 끝났고, 몇 대가 진행 중인지"를 추적하는 용도로 사용합니다.

마이그레이션 준비 단계에서는 **Application Discovery Service**로 온프레미스 서버의 사용률과 의존성을 먼저 수집하는 것이 일반적인 순서입니다. 이 데이터를 기반으로 어떤 서버를 어떤 R 전략으로 분류할지 결정합니다.

## Database Migration Service (DMS)

**DMS**는 데이터베이스를 최소 다운타임으로 이전하는 서비스입니다. 두 가지 핵심 사용 패턴이 있습니다.

- **동종 마이그레이션(Homogeneous)**: 같은 엔진(예: Oracle → Oracle)으로 이전. 스키마 변환이 거의 필요 없음
- **이종 마이그레이션(Heterogeneous)**: 다른 엔진(예: Oracle → Aurora PostgreSQL)으로 이전. 이때는 **Schema Conversion Tool**(SCT)로 스키마와 저장 프로시저를 먼저 변환해야 함

DMS는 지속적 복제(CDC, Change Data Capture)를 지원하므로, 초기 풀 로드 이후에도 원본 DB의 변경 사항을 목표 DB에 실시간으로 반영할 수 있습니다. 이 덕분에 컷오버(전환) 시점의 다운타임을 수 분 이내로 줄일 수 있습니다.

## 모놀리식에서 마이크로서비스로

Refactor 전략의 대표적인 사례가 모놀리식 애플리케이션의 마이크로서비스 전환입니다. 일반적인 접근은 한 번에 전부 바꾸는 것이 아니라 점진적으로 분리하는 것입니다.

1. **Strangler Fig 패턴**: 모놀리식 앞에 API Gateway나 ALB를 두고, 신규 기능부터 하나씩 별도 Lambda/ECS 서비스로 분리해 라우팅
2. **데이터 분리**: 공유 데이터베이스를 서비스별로 분리하기 전, 먼저 읽기 전용 복제본으로 분리 영향을 테스트
3. **이벤트 기반 통신 도입**: 서비스 간 직접 호출 대신 EventBridge나 SQS를 통한 비동기 통신으로 결합도 낮추기

{{< callout type="warning" >}}
마이크로서비스 전환은 기술적 성공만으로 끝나지 않습니다. 서비스 수가 늘어나면 운영 복잡성(모니터링, 배포, 트러블슈팅)도 함께 증가하므로, 전환 전에 CloudWatch, X-Ray 같은 가시성 도구를 먼저 갖추는 것이 권장됩니다. 운영 측면은 [Well-Architected: 운영 우수성](../../well-architected/operational-excellence/)에서 더 다룹니다.
{{< /callout >}}
